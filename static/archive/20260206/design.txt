# Debugger Interface Portfolio ‚Äî Design Specification

## Overall Concept

The entire single-page website is styled to look like an IDE/debugger environment ‚Äî think a dark-themed code editor (VS Code / JetBrains inspired) with a muted charcoal background (#1e1e2e), monospaced fonts (JetBrains Mono or Fira Code), and syntax-highlighting color accents. The page is framed as a single source code file called `geva_kipper_portfolio.life` open in a tab at the top. A thin left-hand gutter displays line numbers running down the entire page. A narrow top bar mimics a window title bar with fake minimize/maximize/close dots (red, yellow, green) on the left and the filename in the center. The scrollbar on the right is custom-styled to look like a code minimap.

## Header / Hero Area

At the very top of the "code file," lines 1‚Äì8 are rendered as a block comment (using `/*` and `*/` delimiters, each line prefixed with `*`). Inside this comment block, the developer's name **Geva Kipper** is displayed in a large, bold monospaced font colored in a bright teal/cyan (#89dceb), styled as if it were a doc-string annotation like `@author Geva Kipper`. Directly below, in a softer lavender (#cba6f7) and slightly smaller font, a single line reads: *"Software Engineer at Google, building large-scale systems that touch billions of users."* This line is formatted as a `@description` tag. Below that, separated by an empty comment line, a `@notice` annotation in a muted yellow/gold (#f9e2af) italic font states: *"This portfolio is redesigned each day by AI ‚Äî what you see today will never appear again."* A subtle CSS animation makes this subtitle line gently pulse in opacity, as though the text itself is alive and ephemeral. The entire comment block has a slightly lighter background strip (#27273a) to set it apart, mimicking how some editors highlight doc blocks.

## The "Bug" Interaction Mechanic

After the comment block, a line of code reads something like `function loadPortfolio() {` in standard syntax coloring (keyword in magenta, function name in blue). Below this opening brace, each of the six project sections is presented as a block of "code" that initially contains a visible **bug** ‚Äî a deliberate, obvious error highlighted with a red squiggly underline and a red circular error icon in the left gutter (mimicking IDE error markers). Each bug is a playful, thematic puzzle: for example, a variable name is misspelled, an operator is wrong, a string is clearly garbled, a semicolon is in the wrong place, a comparison uses `=` instead of `==`, or an array index is off by one. Next to each buggy line, a red badge shows an error count (e.g., "1 error"). The section content below the buggy line is blurred out and overlaid with a translucent pane that says `// CONTENT BLOCKED ‚Äî runtime error on line N` in red italic text. 

When the visitor clicks on the buggy line, a small **inline popup** appears directly at the error location ‚Äî styled exactly like an IDE's quick-fix tooltip with a lightbulb icon and a dropdown showing one or two fix options (e.g., "Change `=` to `==`" or "Fix typo: `projecct` ‚Üí `project`"). When the visitor clicks the correct fix, a satisfying micro-animation plays: the red squiggly disappears, the gutter icon flips from a red circle-X to a green circle-check, the error badge turns into a green "0 errors" badge, and the blurred content below gracefully un-blurs with a quick scale-up ease-out animation, as if the code just compiled successfully. A subtle green flash sweeps across the revealed section. A small console-style toast notification slides in from the bottom-right reading something like `> Build succeeded. Section rendered.` in green monospace text.

If the visitor doesn't want to interact, a small "Skip debugging ‚Üí" link at the top (styled as a toolbar button labeled "Run Without Debugging ‚ñ∂") reveals all sections at once with a cascade animation, so the content is always accessible.

## Section Layout and Structure

Each of the six project sections occupies a "code block" visually indented under the `loadPortfolio()` function. They are separated by blank lines and faint horizontal rules styled as comment dividers (`// ‚Äî‚Äî‚Äî‚Äî‚Äî‚Äî‚Äî‚Äî‚Äî‚Äî‚Äî‚Äî‚Äî‚Äî‚Äî‚Äî`). Each section is structured as follows:

**Section header:** A line that looks like a variable declaration, e.g., `const project_1 = new Project({` in syntax-highlighted colors ‚Äî `const` in magenta (#f38ba8), the variable name in white, `new Project` in teal. The project's name is rendered as a string literal in green (#a6e3a1) with quotes, e.g., `name: "GlglStats"`. This name is large and prominent.

**Section body:** Below the declaration line, the content is formatted as key-value pairs in object literal syntax, each on its own "line" with a line number. The description text appears as a string value in warm green. Tags or technologies appear as an array of strings in orange (#fab387), e.g., `tech: ["Python", "LZ77", "Data Visualization"]`. Links appear as values highlighted in bright blue (#89b4fa) and are actual clickable hyperlinks ‚Äî they are underlined and on hover they glow softly and show a tooltip styled as an IDE "peek definition" mini-window that displays a tiny preview or favicon of the destination.

**Section closing:** Each section ends with `});` on its own line.

## Global Design Details

**Color Palette:** Catppuccin Mocha inspired ‚Äî deep base (#1e1e2e), surface (#313244), overlay (#45475a), text (#cdd6f4), with accents in rosewater, flamingo, pink, mauve, red, maroon, peach, yellow, green, teal, sky, sapphire, blue, and lavender.

**Typography:** JetBrains Mono (loaded from Google Fonts) for all text, maintaining the code editor illusion everywhere. Font sizes vary ‚Äî large for the name, medium for project names, small for descriptions and line numbers. Line height is generous (1.7) to mimic code editor spacing.

**Animations:** All animations are subtle and performant ‚Äî no heavy libraries. CSS transitions for hovers and reveals, `@keyframes` for the pulsing subtitle and decorative elements, and vanilla JavaScript for the stopwatch and bug-fix interactions. When a section is unlocked, it uses a combination of `filter: blur()` transitioning to `blur(0)` and `opacity` transitioning from 0.3 to 1, with a slight `transform: translateY(10px)` to `translateY(0)` for a satisfying reveal.

**Responsiveness:** On mobile, the line number gutter becomes narrower (2 digits wide), the indentation decreases, and the sections stack naturally since they're already vertically laid out. The bug-fix tooltips appear as bottom sheets on mobile rather than inline popups. The title bar dots and minimap scrollbar hide on small screens to save space.

**Progress Indicator:** A thin progress bar at the very top of the page (above the title bar) in gradient from red to green shows how many bugs have been fixed out of six (e.g., starts fully red, ends fully green). Next to it, a small counter reads `Bugs remaining: 6` and decrements with each fix, reaching `Bugs remaining: 0 ‚Äî Build successful! ‚úì` in bright green when all are done.

**Easter Egg:** If a visitor opens the browser's actual developer console, a styled `console.log` message greets them: *"Hey, a fellow debugger! üêõ You're already in the right place. ‚Äî Geva"* in large, colored text using `%c` formatting.
